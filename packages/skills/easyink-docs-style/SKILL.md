# EasyInk 文档风格指南

编写 EasyInk 技术文档时遵循以下约定。每条规则都有可对照的正反例。

## 核心原则

### 1. 对话语气：用"我们"和"你"

把读者当作坐在旁边的同事来写。使用"我们"（项目/团队）和"你"（读者），保持直接对话感。

**正确：** "你可能已经有疑问了——别担心，下面会逐一解释。"
**避免：** "读者在此处可能会产生困惑，后续章节将提供说明。"

使用口语化的表达（别、就行、的话），但不随意。

### 2. 先给代码，后讲道理

每个概念都遵循：一句引入 → 代码示例 → 解释这段代码做了什么 → 补充细节。

代码示例是中心，文字围绕它展开。连续两段纯文字还没出现代码块，就该反思了。读者应该能复制粘贴示例并立刻看到结果。

### 3. 渐进展开，不超前引用

- 先用一句话定义概念。
- 然后用最简单的情况展示。
- 再补充细节、边界情况和替代方案。
- 绝不引用当前页面还没介绍过的概念。
- 深入内容用链接引出，不内嵌到当前段落。

### 4. 预判读者的困惑

在读者可能感到吃力的地方主动安抚：

- "你可能已经有疑问了——别着急。"
- "如果这些概念看起来有点吓人，没关系。"
- "关于 X，目前知道这些就够了。"

给读者明确的"停留许可"——允许他们暂时不完全理解，并在后续章节指路。

### 5. 给出选项，不给结论

当存在多种合理做法时，平等呈现：

- 并排展示替代方案。
- 提供决策框架："如果你想做 X，选 A；如果你想做 Y，选 B。"
- 肯定所有路径："两种方式都能完成你的目标。"
- 让读者决定，你提供的是地图，不是终点。

### 6. 杜绝营销腔

不用"惊艳"、"极致"、"革命性"、"业界领先"等空洞形容词。让代码和结果说话。

快 → 给出数据。简洁 → 用更少的代码行数证明。不宣称，只展示。

### 7. 短段落，高可扫描性

- 每段不超过 3–4 句话。
- 功能列表、选项对比和要点总结用 bullet list。
- 每段突出**一个**关键信息点，而非整句加粗。
- 辅助信息用提示块（`:::tip`、`:::warning`）包裹，不打断正文流。
- 每个 H2/H3 都应该能独立浏览。

### 8. 术语准确，解释平实

用正确的技术术语，但首次出现时立刻用自然语言释义。

**正确：** "指令（Directives）是以 `v-` 开头的特殊属性。"
**避免：** "框架利用基于指令的声明式属性绑定系统。"

对读者既不高估，也不低估。

### 9. 明确的路标

读者任何时候都应该知道三件事：
- 我在哪？（清晰的标题层级）
- 接下来看什么？（章节末尾的内部链接）
- 这跟我有关吗？（"如果你使用 X" 之类的前置说明）

每个标题带锚点 ID。章节之间保持链接引用。

### 10. 以目标为导向

每个章节回答一个问题：读者看完之后能做什么之前做不了的事？

- 先讲能力，再讲机制。
- 章节结尾给出具体下一步（试试看、继续阅读、应用到你的项目）。
- "在线试一下" 或等效操作是最理想的章节收尾。

## 结构模板

### 打开一个主题

```
## X 是什么？

X 是 [一句话平实定义]。

看一个最小示例：

[代码块]

上面演示了 [读者刚看到的内容]。
```

### 呈现选择

```
## 该选哪个？

两种方式都能完成你的目标，它们基于同一套底层系统。

- 方式 A：更适合 [条件]。原因：[一句话]。
- 方式 B：更适合 [条件]。原因：[一句话]。

如果你刚开始，选那个看起来更顺眼的。随时可以切换到另一种。
```

### 引入注意事项

```
:::warning 注意
[一句话或两句话说明风险或限制]
:::
```

安全警告、破坏性变更或常见陷阱不要藏在正文中。

### 收尾一个章节

```
关于 X，目前知道这些就够用了。熟悉之后可以阅读 [X 完整指南](/link)。
```

## 避免清单

- 被动语态（"建议使用" → "我们建议你使用"）。
- 将来时承诺（"这将稍后介绍" → 直接给出"详见 [X](/link)"）。
- "只要"、"不过就是"、"显然"、"说白了" 等轻视读者困惑的词。
- 学术腔（"本章我们将探讨"、"读者可"、"一般认为"）。
- 没有可见结果的步骤说明——每一步都要有代码块或可操作的验证点。
- 大片文字没有代码示例穿插。
- 任何营销式最高级形容词。
